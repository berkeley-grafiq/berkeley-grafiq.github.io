Abstract

VLMs are getting good at telling us what is in an image. But can they explain why the image looks that way?

This question matters more than ever because VLMs are increasingly used as judges for image generation: to verify outputs, compare models, and critique rendered results. A model that only understands semantics may still miss the graphics structure that actually determines appearance.

GRAFIQ is a benchmark for testing whether VLMs understand the graphics factors behind image formation, not just image-level semantics. Given rendered images, we ask whether a model can reason about the latent scene properties that produced them.

We focus on four core benchmark aspects:

  • Geometry — Can the model reason about 3D shape, depth, and surface normals?
  • Materials — Can it distinguish albedo from gloss, texture from reflectance, and recognize view-dependent appearance?
  • Lighting — Can it understand shadows, shading, and light direction as causes of visual change?
  • Viewing & Correspondence — Can it track consistency across camera changes, multi-view renderings, and object correspondences?

GRAFIQ is designed to measure whether VLMs understand not only what they see, but also the rendering structure that makes it visible.

Benchmark Construction

Our goal is to build GRAFIQ, a graphics-oriented benchmark that tests whether VLMs understand the factors behind image formation, rather than only recognizing image semantics. To make the benchmark concrete and feasible, we use a controlled synthetic rendering pipeline.

The benchmark is built in three main steps:

1

Create Scenes

We build a set of simple but diverse 3D scenes with varied objects, materials, layouts, and lighting. This can be done with pre-existing publicly available datasets. The goal is to make our dataset as diverse as possible.

2

Vary One Factor at a Time

For each scene, we change only one graphics factor while keeping the others fixed. This lets us isolate the cause of a visual difference. This can be done using Blender or programmatic rendering.

3

Convert Renderings into QA Tasks

Using renderer metadata, we automatically generate questions and labels, then filter out ambiguous cases.

We organize the benchmark into four task categories:

 Geometry

Tests whether the model can reason about 3D structure such as shape, depth, and surface orientation.

  • "Which object is closer to the camera, the red sphere or the blue cube?"
  • "Which surface is more front-facing to the viewer?"
 Materials

Tests whether the model can distinguish intrinsic surface properties such as albedo, glossiness, and reflectance.

  • "Which object has a shinier material, the left mug or the right mug?"
  • "Did this appearance change come from the material or from the lighting?"
 Lighting

Tests whether the model understands illumination as the cause of shading and shadows.

  • "Is the light source coming from the left or the right?"
  • "Which object is casting the shadow on the table?"
 Viewing & Correspondence

Tests whether the model can reason across viewpoint changes and match objects between views.

  • "Do these two images show the same scene from different viewpoints?"
  • "Which object in the second image corresponds to the striped cube in the first image?"

Evaluation Protocol

For each of the four categories, we build tasks in several formats:

  • Binary questions (yes / no)
  • Multiple-choice questions
  • Image-pair or multi-view comparison questions

The main evaluation metric is task accuracy. We report:

  • Overall accuracy across the full benchmark
  • Per-category accuracy for geometry, materials, lighting, and viewing/correspondence
  • Breakdowns by task type and scene variation

This helps us see not only whether a model performs well overall, but also which graphics factors it understands well and which remain difficult. If feasible, we will also include a small human baseline for comparison.

Deliverables

What We Plan to Deliver

  • A controlled synthetic data generation pipeline for graphics-based benchmarking
  • A benchmark dataset with tasks covering geometry, materials, lighting, and viewing & correspondence
  • Automatically generated ground-truth labels from renderer metadata
  • An evaluation of several existing VLMs on the benchmark
  • Quantitative results with fine-grained breakdowns by category and task type

What We Hope to Deliver

  • A clearer diagnosis of how current VLMs fail on graphics-related reasoning
  • An error analysis or taxonomy of common failure modes
  • A small human baseline for reference
  • A simple baseline method or prompting strategy that improves performance or interpretability

Resources

Related Works

  • BLINK: Multimodal Large Language Models Can See but Not Perceive — Shows that strong VLMs can still fail on basic visual perception tasks, which motivates our graphics-focused benchmark.
  • PhysBench: Benchmarking and Enhancing Vision-Language Models for Physical World Understanding — A benchmark studying whether VLMs understand physical properties, relationships, scenes, and dynamics, helping us position our work as focusing on graphics-grounded visual reasoning.

Computing Platform, Hardware, and Software

We will use our local machines as the main computing platform. Our software stack will mainly consist of Blender for rendering and scene generation, Python for data processing and benchmark construction, and PyTorch or Hugging Face Transformers for running and evaluating vision-language models.

Starting Point

We are not starting from a full existing benchmark codebase. Instead, we will build our own synthetic data-generation and evaluation pipeline using Blender scripting and standard VLM inference libraries. We may also incorporate publicly available 3D assets or scene datasets depending on the need.
